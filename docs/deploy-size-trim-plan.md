# Deploy Size Trim Plan

Related: [Render Optimization Plan](render-optimization-plan.md), [Scale-Up Plan](scaleup-plan.md).

## Purpose

Reduce the GitHub Pages deploy payload from 9.48 MB gzipped to something smaller without breaking the interactive map. Identify which trims are free (no visible degradation) versus which trade fidelity for bytes, and recommend a default before cutting the first public deploy.

Mirrors `render-optimization-plan.md` in format, but targets the static deploy artifacts (`docs/*.geojson`, `docs/*.png`) rather than the client render path. Picks up on the "reduce coordinate precision" idea flagged in `scaleup-plan.md` Risks table.

## Scope

In scope.
- Coordinate precision reduction (round lon/lat to fewer decimal places).
- Geometric simplification (Douglas-Peucker) at multiple tolerances.
- Initial-only deploy (drop full `*.geojson`, keep only MIT-viewport chunks).

Out of scope this pass.
- TopoJSON and vector tiles. Require client-side library change (topojson-client or MapLibre VectorGrid), bigger than a size trim.
- Property pruning. All three properties in use. `BLDG_HGT_2010` drives building color, `height_ft` drives shadow color, `shadow_len_ft` is in the click popup (`docs/prototype.html` lines 164, 170-171, 264, 267).
- Pre-compressed `.gz` sidecars. GitHub Pages applies on-the-fly gzip, but cannot attach the right `Content-Encoding` header to arbitrary `.gz` files. Pre-compressed sidecars are therefore dropped from the deploy set entirely.

## Baseline

Generated by `src/prototype.py --scale 100` (123,105 buildings plus 123,105 shadow polygons, server-side simplified at 3 m per render strategy r10).

| File | Raw | Gzipped (level=6) | Features | Vertices |
|------|----:|------------------:|---------:|---------:|
| `shadows.geojson` | 29.24 MB | 4.09 MB | 123,105 | 635,633 |
| `buildings.geojson` | 26.47 MB | 4.22 MB | 123,105 | 605,054 |
| `shadows_initial.geojson` | 3.60 MB | 0.52 MB | 14,852 | 79,654 |
| `buildings_initial.geojson` | 3.23 MB | 0.50 MB | 14,795 | 74,928 |
| `shadows.png` | 0.15 MB | 0.15 MB | — | — |
| **Total** | **62.70 MB** | **9.48 MB** | | |

Gzipped totals are what users actually download. GitHub Pages negotiates `Content-Encoding: gzip` on text responses. `shadows.png` is already compressed so gzip is a no-op there.

## Methodology

Scripts (exploration pass, live in `/tmp` for now. Move to `scripts/` if the recommendation ships).

- `/tmp/trim_experiment2.py` — rounds coords and/or Douglas-Peucker simplifies every file, reports raw plus `gzip.compress(level=6)` size and vertex count per variant.
- `/tmp/generate_variants.py` — writes each variant to `/tmp/variants/<variant>_<file>` so downstream parse and wire benches can load them.
- `/tmp/parse_bench.mjs` — Node 25 `JSON.parse` best-of-5 per variant, 2-run warmup. N=5 >= 3 from `.claude/CLAUDE.md` bench rule.

All gzip sizes use level 6 to match GitHub Pages' on-the-fly setting, not Python's default level 9 (which under-reports wire cost by ~2 percent).

## Results

### Precision reduction

At latitude 42 degrees.

| Decimals | Precision | Visual impact |
|---|---|---|
| 6 (current) | 11 cm | Overkill |
| 5 | 1.1 m | Below one display pixel at max zoom |
| 4 | 11 m | Noticeable stepping on building outlines. Soft shadow blobs tolerate better |
| 3 | 110 m | Unacceptable. Buildings disappear |

Per-file gzipped delta vs baseline.

| File | prec=5 | prec=4 | prec=3 |
|------|-------:|-------:|-------:|
| `shadows.geojson` | -17.8% | -40.9% | -69.7% |
| `buildings.geojson` | -15.5% | -38.8% | -65.7% |
| `shadows_initial.geojson` | -17.9% | -41.3% | -69.3% |
| `buildings_initial.geojson` | -16.8% | -39.5% | -68.2% |

Gzip gets bigger gains than raw because fewer unique digit patterns compress better.

### Simplification (Douglas-Peucker)

Server-side pipeline already simplifies at 3 m (strategy r10 in `render-optimization-plan.md`). Further client-side simplification buys little.

| Tolerance | shadows gz Δ | buildings gz Δ | vertex drop |
|---|---:|---:|---:|
| 0.5 m | -1.0% | 0.0% | 1-2% |
| 1.0 m | -1.3% | 0.0% | 1-2% |
| 2.0 m | -1.8% | 0.0% | 2-3% |
| 5.0 m | -2.6% | -0.1% | 3-4% |
| 10.0 m | -13.2% | -11.8% | 13-16% |

Only 10 m moves the needle, and at that tolerance building corners visibly round off. Precision=5 reaches similar savings with no corner loss.

### Aggregate deploy scenarios

| Scenario | raw | gz | Δ gz |
|---|---:|---:|---:|
| baseline | 62.70 MB | 9.48 MB | — |
| prec=5 everywhere | 59.91 MB | 7.92 MB | -16.5% |
| prec=5 buildings + prec=4 shadows (**asym**) | 58.48 MB | 6.86 MB | **-27.7%** |
| prec=4 everywhere | 57.12 MB | 5.76 MB | -39.3% |
| simp=1m + prec=5 everywhere | 59.64 MB | 7.88 MB | -16.9% |
| initial-only (drop full geojson) | 6.99 MB | 1.16 MB | -87.7% |

### Parse-time impact

Node.js `JSON.parse` best-of-5, 2-run warmup.

| Variant | initial (ms) | full (ms) | total (ms) |
|---|---:|---:|---:|
| baseline | 36.4 | 369.5 | 405.9 |
| prec=5 | 35.4 | 361.7 | 397.1 |
| prec=4 | 37.4 | 355.5 | 392.9 |
| asym | 37.5 | 355.4 | 392.9 |

Parse time varies within noise (~3 percent). Savings are **entirely in wire cost**, not parse cost. V8 parses all variants at roughly the same speed. Character count drops only 5-10 percent even when gzip drops 30 percent.

### End-to-end wire time projection

Using gzipped sizes and two reference downlinks, plus a constant ~0.4 s parse.

| Variant | gz (MB) | 10 Mbps | 100 Mbps |
|---|---:|---:|---:|
| baseline | 9.48 | ~8.0 s | ~1.16 s |
| prec=5 | 7.92 | ~6.7 s | ~1.03 s |
| asym | 6.86 | ~5.9 s | ~0.95 s |
| initial-only | 1.16 | ~1.3 s | ~0.49 s |

On a 10 Mbps link the `asym` variant saves roughly 2 seconds of perceived full-map ready time. On fiber the absolute saving is small, but the relative total time still drops ~20 percent.

## Trade-offs

| Option | Bytes saved (gz) | Visual risk | Code change | Verdict |
|---|---:|---|---|---|
| baseline | 0 | none | none | — |
| prec=5 everywhere | -16.5% | none (1.1 m below one pixel) | `src/prototype.py` write-precision arg | **Recommended** (visual diff confirmed invisible). See "Recommendation (revised)" below |
| asym (b=5, s=4) | -27.7% | low. Shadows are soft blobs. 11 m step may be invisible vs the ~3 m server simplify already in place | same | Defensible second pick if 1.5 MB extra matters. Visual diff shows mild shadow-edge shift |
| prec=4 everywhere | -39.3% | high. Building outlines visibly stepped at high zoom | same | Skip |
| simp 10 m | -13.2% | medium. Lost corners | same | Skip. Precision=5 beats it without corner loss |
| initial-only | -87.7% | pan outside MIT -> empty map layer | `docs/prototype.html` skip full fetch. Keep `shadows.png` as coarse fallback | Secondary flavor for a low-bandwidth demo URL |

## Browser Render Bench (2026-04-18)

Real end-to-end bench using `scripts/render_verify.py` (Playwright headless Chromium) against `scripts/serve.py` on localhost:8765, with each variant swapped into `docs/` just before the run. N=3 per variant plus a discarded warmup run at the start (cold DNS/TCP/CDN state pollutes the first request).

Orchestrator `/tmp/run_deploy_bench_v2.py`, result summary `benchmarks/deploy-size-trim/bench_v2_warmup.json`.

### Median-of-3 timings, post-warmup (ms)

| Variant | fetchEnd | initialAt | addedAt | LCP | FCP | shadow.gz |
|---------|---------:|----------:|--------:|----:|----:|----------:|
| baseline | 2013 | 660 | **2805** | 792 | 376 | 4.09 MB |
| prec5 | 3941 | 1390 | 4905 | 1544 | 548 | 3.36 MB |
| asym | 2586 | 852 | 3185 | 1104 | 436 | 2.42 MB |
| prec4 | 2583 | 980 | 3033 | 1028 | 376 | 2.42 MB |
| empty | 919 | 495 | **919** | 916 | 432 | 58 B |

### Important caveat — localhost bench does not measure wire savings

Every fetch is from loopback (~100 Gbps). A 4 MB vs 2 MB sidecar download takes <1 ms vs <0.5 ms on loopback, both invisible. So this bench captures **browser parse + Leaflet layer build cost, not wire transfer cost.**

Within-variant variance (WSL2 host noise) is ~2x — consistent with the 2-5x drift called out in `bench-protocol.md`. That noise floor swamps the parse-cost signal between `baseline`, `prec5`, `asym`, `prec4`, which are all within ~2 MB of each other. Only `empty` drops far enough (~4 MB to 58 bytes) to show a clear speedup (`addedAt` 2805 -> 919 ms, a ~3x reduction).

**Do not interpret this table as "prec5 is slower than baseline".** Those numbers are noise. The authoritative savings metric is the gzipped byte count (table above in the Aggregate deploy scenarios section), not localhost timings.

### First-bench cold-start artifact

An earlier bench (`bench_v1_cold-baseline.json`) ran without a warmup, so `baseline_r0` paid the first-of-session CDN cost (6394 ms addedAt vs ~3000 ms for later runs) and every other variant, which ran after that warm-up point, looked artificially faster relative to baseline. Do not cite the v1 numbers. Kept on disk only for the cold-start contrast.

## Recommendation (revised)

**Default: precision=5 everywhere. Saves 16.5 percent gzipped with zero visible difference.** The earlier tentative recommendation (asym) saves a further 11 points but trades mean pixel Δ 1.5 -> 5.9 and max Δ 148 -> 172. For a class project demo, the extra 1.5 MB off a 10 Mbps download (~1 second) is not worth a visible quality dip. For a latency-critical public tool, asym would still be a defensible call after an explicit sign-off.

**Secondary deploy: empty (initial-only).** Offer a `?fast=1` query-param or a separate `index-fast.html` that suppresses the full fetch. 87.7 percent smaller on the wire, full MIT-area interactivity retained. The obvious UX loss (pan outside MIT -> only PNG preview, no clickable shadows) is acceptable for a "just want to see it" landing page.

**Do not ship prec=4.** Building outlines visibly stepped.
